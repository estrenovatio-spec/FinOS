# RFC-003 Income Sources for Safe Spending

## Статус

Draft

## Контекст

Money Setup v1 уже реализован и хранит минимальную финансовую базу:

- `nextIncomeDate`
- `expectedIncomeAmount`
- `useHouseholdBalance`
- `requiredRecurringIds`
- `essentialCategoryIds`

Этого достаточно для простого сценария, где у пользователя есть один ближайший доход.

Но Safe Spending не должен опираться только на этот кейс.

## Проблема

Текущая single-income model предполагает, что пользователь живёт до одной ближайшей выплаты.

Это плохо покрывает реальные сценарии:

- аванс + зарплата;
- несколько работодателей;
- аренда как отдельный доход;
- подработка;
- бизнес;
- пассивный доход.

В таких случаях одна пара полей:

- `nextIncomeDate`
- `expectedIncomeAmount`

слишком грубо описывает финансовую реальность пользователя.

Риск:

FinOS начнёт считать Safe Spending слишком жёстко или слишком мягко, а значит даст ложную точность.

## Цель RFC

Определить минимальное расширение Money Setup для нескольких доходов без усложнения текущего UX.

## Почему single-income model недостаточна

### 1. Две выплаты в месяц

У многих пользователей есть:

- аванс;
- зарплата.

Если FinOS знает только одну дату и одну сумму, он не понимает:

- что до зарплаты может быть ещё аванс;
- что горизонт риска короче, чем кажется;
- что пользователь живёт не до конца месяца, а до ближайшей из нескольких выплат.

### 2. Несколько источников дохода

Часто деньги приходят из разных источников:

- работа по найму;
- аренда;
- подработка;
- бизнес;
- пассивный доход.

Один `expectedIncomeAmount` схлопывает эти источники в одно число и скрывает их разную дату поступления.

### 3. Safe Spending зависит не только от суммы, но и от времени

Для расчёта безопасного лимита важно понимать:

- когда именно придут деньги;
- сколько именно ожидается;
- сколько промежутков между доходами надо пережить.

Single-income model хранит только один ближайший снимок и не даёт достаточной опоры для более честного расчёта.

## Proposed Model

### Минимальная сущность `MoneySetupIncomeSource`

```ts
type MoneySetupIncomeSource = {
  id: string
  label: string
  expectedDate: string | null
  expectedAmount: number | null
  kind: "salary" | "advance" | "rent" | "freelance" | "business" | "passive" | "other"
  isPrimary?: boolean
}
```

## Почему именно такая модель

Она решает минимальную задачу:

- хранит несколько источников;
- хранит дату;
- хранит сумму;
- позволяет отличать зарплату от аванса и аренды;
- не требует сложной периодической логики.

## Как `incomeSources` живёт внутри `MoneySetup`

```ts
type MoneySetup = {
  nextIncomeDate: string | null
  expectedIncomeAmount: number | null
  incomeSources?: MoneySetupIncomeSource[]
  useHouseholdBalance: boolean
  requiredRecurringIds: string[]
  essentialCategoryIds: string[]
  updatedAt: string | null
}
```

## Совместимость с v1

### Старые поля остаются

Нужно сохранить:

- `nextIncomeDate`
- `expectedIncomeAmount`

Они остаются:

- fallback для старых пользователей;
- single-income mode для простого setup;
- временной опорой, пока UI и Safe Spending не научатся работать с массивом источников.

### Правило совместимости

Если `incomeSources` отсутствует или пуст:

- FinOS работает по старой модели.

Если `incomeSources` заполнен:

- Safe Spending в будущем должен использовать `incomeSources` как основной источник правды;
- старые поля могут считаться legacy-совместимостью и derived summary.

### Зачем не удалять старые поля сразу

- чтобы не ломать sync;
- чтобы не ломать текущий MoneySetupDialog;
- чтобы не требовать миграции UX в один шаг;
- чтобы сохранить мягкий переход от v1 к v2.

## Почему не делаем сложный `incomeSchedule` с recurrence в v1

Сейчас не нужно делать:

- cron-like расписание;
- frequency rules;
- payday templates;
- auto-rollover по месяцам;
- прогноз нерегулярных доходов.

Причины:

1. Это резко усложняет модель.
2. Это усложняет UI и onboarding.
3. Это выглядит точнее, чем есть на самом деле.
4. Для первого Safe Spending достаточно явного списка ожидаемых доходов.

`incomeSources` лучше подходит для v1.5/v2:

- он проще;
- понятнее пользователю;
- легче хранится в JSON;
- легче синхронизируется;
- легче нормализуется.

## UX-принцип

UI должен оставаться простым.

### По умолчанию

Показываем пользователю тот же короткий setup:

1. Следующий доход
2. Сумма дохода
3. Обязательные платежи
4. Необходимые категории

### Позже

Можно добавить вторичный путь:

`У меня несколько выплат`

или

`Добавить ещё источник дохода`

Важно:

- не открывать эту сложность всем пользователям сразу;
- не расширять текущий `MoneySetupDialog` прямо сейчас;
- не превращать setup в анкету бухгалтера.

## Как это повлияет на будущий Safe Spending

После расширения модели Safe Spending сможет:

- точнее определять ближайший доход;
- понимать, что между сегодня и концом месяца может быть не одна, а несколько выплат;
- учитывать доходы по датам, а не как одно усреднённое число;
- строить более честный горизонт расчёта.

Это не означает, что сразу нужно делать сложную формулу.

Но без `incomeSources` даже простая формула будет слабой для пользователей с:

- авансом и зарплатой;
- несколькими потоками дохода.

## Что НЕ делать

В рамках этого расширения не нужно:

- делать cron;
- делать прогноз доходов;
- делать `confidence` или `reliability`;
- делать банковский импорт;
- делать автоопределение доходных расписаний;
- усложнять текущий `MoneySetupDialog` прямо сейчас;
- переписывать весь Money Setup flow.

## Решение RFC

Перед реализацией Safe Spending нужно расширить `MoneySetup` полем:

- `incomeSources`

При этом:

- текущие поля `nextIncomeDate` и `expectedIncomeAmount` остаются;
- новая модель должна быть backward-compatible;
- UI можно усложнять постепенно и только после data layer решения.

## Итоговая рекомендация

Safe Spending не стоит строить только на single-income model, если продукт хочет избежать ложной точности.

Минимально правильный порядок:

1. Утвердить RFC-003.
2. Расширить `MoneySetup` data model полем `incomeSources`.
3. Сохранить совместимость с `nextIncomeDate` и `expectedIncomeAmount`.
4. Только потом переходить к Safe Spending calculation.
