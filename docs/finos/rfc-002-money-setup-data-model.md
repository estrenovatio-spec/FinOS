# RFC-002 Money Setup Data Model

## Статус

Draft

## Контекст

FinOS уже умеет показывать текущий остаток денег, но для честного Safe Spending Limit этого недостаточно.

Чтобы посчитать безопасную сумму на сегодня, системе нужно понимать:

- когда будет следующий доход;
- сколько денег ожидается;
- какие расходы точно нужно закрыть до этого момента;
- какие категории трат являются жизненно необходимыми, даже если у них нет точной даты.

## Проблема

Не все обязательные расходы одинаковы.

Одна часть расходов хорошо описывается как платёж с датой и суммой.

Примеры:

- аренда;
- коммуналка;
- школа;
- ипотека;
- кредиты;
- связь.

Другая часть обязательных расходов не привязана к одной точной дате, но без неё нельзя честно оценить безопасный лимит.

Примеры:

- продукты;
- транспорт;
- здоровье;
- дети;
- бензин.

Если смешать эти два типа в одну модель, FinOS либо начнёт считать слишком жёстко, либо будет давать ложную точность.

## Цель RFC

Определить минимальную модель `moneySetup` для v1, которая:

- не ломает текущую синхронизацию;
- поддерживает fixed obligations;
- поддерживает essential variable spending;
- остаётся достаточно простой для короткого setup flow.

## Предлагаемая модель

```ts
type MoneySetup = {
  nextIncomeDate?: string | null
  expectedIncomeAmount?: number | null
  useHouseholdBalance?: boolean
  requiredRecurringIds?: string[]
  essentialCategoryIds?: string[]
}
```

## Значение полей

### `nextIncomeDate`

Дата следующего ожидаемого дохода.

Нужна, чтобы понимать горизонт расчёта: до какого дня FinOS должен распределять остаток денег.

### `expectedIncomeAmount`

Оценка суммы следующего дохода.

В v1 это простое число без сложной модели confidence, source и frequency.

### `useHouseholdBalance`

Нужно ли учитывать общий семейный баланс, а не только личную часть пользователя.

### `requiredRecurringIds`

Список `recurringTransactions`, которые пользователь считает обязательными fixed obligations.

Это платежи:

- с понятной датой;
- с ожидаемой суммой;
- которые нужно закрыть до следующего дохода или конца расчётного периода.

Примеры:

- аренда;
- коммуналка;
- школа;
- ипотека;
- минимальные платежи по кредитам;
- связь.

### `essentialCategoryIds`

Список категорий, которые пользователь считает essential variable spending.

Это не один конкретный платёж, а необходимый тип трат без точной даты списания.

Примеры:

- продукты;
- транспорт;
- здоровье;
- дети;
- бензин.

## Почему нужен `essentialCategoryIds`

Safe Spending Limit нельзя считать честно только по фиксированным платежам.

Если вычесть из остатка только аренду и кредиты, система может показать слишком высокий “безопасный” лимит, игнорируя базовые ежедневные траты.

`essentialCategoryIds` нужны, чтобы:

- отделить обязательные жизненные категории от необязательных;
- учитывать переменные, но необходимые траты до следующего дохода;
- строить более честную формулу Safe Spending Limit без сложного бюджета v1.

## Чем `essentialCategoryIds` отличаются от `requiredRecurringIds`

`requiredRecurringIds`:

- привязаны к конкретным recurring items;
- имеют дату;
- обычно имеют ожидаемую сумму;
- подходят для fixed obligations.

`essentialCategoryIds`:

- привязаны к категориям, а не к отдельным платежам;
- не требуют конкретной даты;
- могут иметь плавающую сумму;
- подходят для essential variable spending.

Иначе говоря:

- аренда раз в месяц — это `requiredRecurringIds`;
- продукты каждый день или несколько раз в неделю — это `essentialCategoryIds`.

## Как `essentialCategoryIds` могут использовать `categoryBudgets`

Если у категории уже есть `categoryBudget`, FinOS в будущем может использовать этот лимит как ориентир для essential spending.

Пример:

- категория `groceries` помечена как essential;
- у неё есть месячный `categoryBudget`;
- система может взять остаток или пропорциональную часть этого бюджета до следующего дохода.

Это даёт мягкую и понятную опору без необходимости строить отдельную сложную модель переменных обязательных расходов.

## Почему нельзя автоматически считать все `categoryBudgets` обязательными

Не каждый бюджет означает жизненную необходимость.

Пользователь может поставить бюджет на:

- подарки;
- отдых;
- кафе;
- одежду;
- хобби.

Если автоматически считать все `categoryBudgets` обязательными, Safe Spending Limit станет:

- слишком консервативным;
- нечестным;
- визуально пугающим;
- непредсказуемым для пользователя.

Поэтому решение “категория важна для безопасного лимита” должно быть явным пользовательским выбором через `essentialCategoryIds`, а не побочным эффектом наличия бюджета.

## Влияние на будущий Safe Spending Limit

В будущей формуле Safe Spending Limit v1 можно будет опираться на три слоя:

1. Текущий доступный баланс.
2. Fixed obligations из `requiredRecurringIds`.
3. Essential variable spending из `essentialCategoryIds`, при наличии с опорой на `categoryBudgets` или историю категории.

Это позволит считать лимит аккуратнее:

- сначала вычитать обязательные фиксированные платежи;
- затем резервировать базовые необходимые категории;
- и только потом показывать остаток как “безопасно сегодня”.

## Ограничения v1

В v1 не решаем:

- автоматическую классификацию essential categories;
- сложный прогноз трат по дням;
- автоматическое назначение обязательности всем recurring;
- автопривязку всех budgets к safe spending;
- сложные правила для разных участников семьи.

## UX-следствие

Money Setup v1 может остаться коротким:

1. Когда следующий доход?
2. Сколько примерно придёт?
3. Какие регулярные обязательные платежи учитывать?
4. Какие жизненно важные категории учитывать?
5. Считать по личному или семейному балансу?

Это всё ещё компактный setup, без большого отдельного flow.

## Риски

### Риски `requiredRecurringIds`

- ссылки могут устареть, если recurring удалён;
- список нужно чистить при replace/import planning;
- recurring item может остаться в списке, хотя больше не актуален.

### Риски `essentialCategoryIds`

- пользователь может отметить слишком много категорий;
- без бюджета или истории оценка по категории будет приблизительной;
- разные семьи по-разному понимают “необходимые траты”.

## Решение для v1

Для v1 это acceptable tradeoff:

- `requiredRecurringIds` покрывают fixed obligations;
- `essentialCategoryIds` покрывают essential variable spending;
- `categoryBudgets` используются как дополнительная опора, но не как автоматический признак обязательности.

Это даёт достаточно честную и при этом простую основу для EPIC-009 и EPIC-007, не требуя новой сложной финансовой модели.
