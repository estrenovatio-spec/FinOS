# EPIC-007 Safe Spending Limit

## Статус

Draft

## Цель

Показать пользователю не только сколько денег есть сейчас, а сколько можно тратить безопасно.

FinOS должен отвечать не на вопрос:

Сколько у меня денег?

а на вопрос:

Сколько я могу потратить, чтобы не развалить месяц?

## Проблема

Текущий статус “Доступно сейчас” показывает остаток.

Но остаток на счёте не равен безопасной сумме для трат.

Пример:

На балансе 96 000 ₽.

Но если до зарплаты 21 день, впереди аренда, школа, еда и транспорт, то безопасно сегодня может быть не 96 000 ₽, а 2 300 ₽.

## Главная идея

Разделить два понятия:

1. Доступно сейчас — текущий свободный остаток.
2. Безопасно сегодня — дневной лимит с учётом периода и обязательных расходов.

## User Story

Как пользователь,  
я хочу понимать, сколько могу потратить сегодня без вреда,  
чтобы не жить в тревоге и не считать всё в голове.

## Scope v1

Входит:

- провести диагностику текущих данных;
- понять, есть ли в store регулярные расходы;
- понять, есть ли period/payday logic;
- понять, есть ли planned payments;
- понять, как сейчас считается главный статус;
- предложить минимальную формулу v1;
- не менять UI до диагностики.

## Out of Scope v1

Не входит:

- сложное бюджетирование;
- AI-прогноз;
- банковские интеграции;
- подписки;
- полноценный календарь платежей;
- новая модель данных без RFC;
- синхронизация.

## Возможная формула v1

safeToday = max(0, (availableBalance - requiredUntilNextIncome) / daysUntilNextIncome)

Где:

- availableBalance — текущий доступный остаток;
- requiredUntilNextIncome — обязательные расходы до следующего дохода;
- daysUntilNextIncome — дней до следующего дохода или конца периода.

## Если данных мало

Если нет регулярных расходов или даты дохода, не показывать ложную точность.

Вместо этого:

Доступно сейчас: 96 000 ₽  
Для безопасного лимита добавьте регулярные расходы и дату дохода.

## UX-принципы

1. Не обманывать точностью.
2. Не говорить “можно потратить”, если это просто остаток.
3. Разделять остаток и безопасный лимит.
4. Показывать расчёт только если есть достаточно данных.
5. Сначала доверие, потом сложность.

## Acceptance Criteria для диагностики

- [ ] Найдено, как считается текущий главный статус Today.
- [ ] Найдены источники баланса.
- [ ] Найдены регулярные расходы, если они есть.
- [ ] Найдены planned payments, если они есть.
- [ ] Найдена логика доходов/payday, если она есть.
- [ ] Предложена формула v1.
- [ ] Описаны данные, которых не хватает.
- [ ] Код не менялся без отдельного решения.
