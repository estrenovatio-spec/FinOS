# FIN OS 2 — Sprint 1 Implementation Plan

## 1. Какие части проекта уже подходят под FIN OS 2

- `src/components/TodayScreen.tsx` уже является главным экраном и содержит базу для нового центра приложения.
- `src/components/VoiceRecorder.tsx` уже умеет принимать обычный человеческий текст и быстро создавать операции.
- `src/store/useStore.ts` уже содержит основную доменную модель: операции, регулярные операции, долги, цели, лимиты, money setup, семейный режим.
- `src/lib/safe-spending.ts` уже считает безопасный дневной лимит и подходит как база для блока “Хватит ли мне денег?”.
- `src/hooks/useRecurringProcessor.ts` и логика pending recurring уже подходят для блока “Сегодня оплатить”.
- `src/components/TransactionList.tsx` уже решает отображение истории операций и подходит для вкладки “Операции”.
- `src/components/MoneySetupDialog.tsx` уже даёт основу для данных о ближайшем доходе и обязательных расходах.
- `src/lib/ai.ts`, `src/lib/parse-voice-server.ts`, `src/lib/ai-memory.ts`, `src/lib/planning/parse-input.ts` уже закрывают текущую AI-логику и быстрый ввод.

## 2. Какие компоненты можно переиспользовать

- `TodayScreen` как существующую точку входа, но с новым составом блоков.
- `VoiceRecorder` как ядро для кнопки “Добавить операцию”.
- `PendingRecurringCard` частично: его логика подтверждения платежей полезна для нового блока “Сегодня оплатить”.
- `TransactionList` для вкладки “Операции”.
- `MoneySetupDialog` для сценариев, где не хватает данных для ответа “Хватит ли мне денег?”.
- `Button`, `Card`, `Dialog` и текущую UI-библиотеку без добавления новых зависимостей.
- Zustand store и существующие selectors/hooks вместо нового data layer.

## 3. Какие компоненты необходимо переписать

- `src/components/TodayScreen.tsx`
  Нужно не удалять экран, а переориентировать его под 8 новых блоков Sprint 1.
- `src/components/app/AppBottomNav.tsx`
  Текущие табы отражают старую структуру (`Дом`, `Цели и планы`, `Финсоветник`, `Биз`, `Ещё`).
- `src/lib/app-bottom-nav.ts`
  Нужны новые идентификаторы вкладок, иначе появится технический долг в виде “старые id / новые смыслы”.
- `src/app/page.tsx` и `src/components/app/PreviewAppShell.tsx`
  Нужно заменить композицию экранов под новую навигацию, но без переписывания всего shell.

## 4. Какие новые компоненты нужно создать

- `TodayOverviewCard` или аналогичный thin-компонент/утилиту для сборки ответов на 5 главных вопросов пользователя.
- Компонент блока “Сегодня оплатить” поверх существующей pending recurring логики.
- Компонент блока “Следующий риск”.
- Компонент блока “Главное действие”.
- Компонент блока “Сегодня можно”.
- Компонент/диалог для сценария “Добавить операцию” с возвратом на экран “Сегодня”.
- Лёгкие tab screens или wrappers для новых разделов:
  - “Операции”
  - “Прогноз”
  - “Регулярные операции”
  - “Настройки”

## 6. Какие риски я вижу

- В проекте уже есть незакоммиченные изменения в `TodayScreen`, `safe-spending`, `useStore` и связанных файлах. Нужно встраиваться поверх них, не перетирая текущую работу.
- Нижняя навигация сейчас завязана на старые продуктовые сущности. Простое переименование подписей без изменения ids даст быстрый результат, но закрепит путаницу в коде.
- `PlanningPanel` содержит слишком много логики сразу. Если попытаться сразу превратить его в “Прогноз” и “Регулярные операции”, можно случайно запустить скрытый рефакторинг.
- `VoiceRecorder` сейчас не сообщает экрану о завершении успешного сохранения. Для UX “сохранить и вернуться на Сегодня” понадобится аккуратное расширение API компонента.
- Часть старого функционала (`Финсоветник`, `Биз`, сервисы, referral`) не входит в Sprint 1, но уже существует. Нельзя просто потерять к нему доступ без явного решения.

## 7. Предлагаемый порядок реализации

1. Зафиксировать новую навигационную модель Sprint 1 без удаления store и backend-логики.
2. Вынести вычисление состояния дня в отдельный helper, чтобы не раздувать `TodayScreen`.
3. Пересобрать `TodayScreen` под 8 обязательных блоков и 5 пользовательских вопросов.
4. Добавить сценарий “Добавить операцию” через существующий `VoiceRecorder` с возвратом на `Сегодня`.
5. Собрать новые вкладки:
   - “Операции” на базе истории операций
   - “Прогноз” на базе существующих planning/forecast данных
   - “Регулярные операции” на базе recurring/pending логики
   - “Настройки” на базе существующего settings/more
6. Спрятать устаревшие для Sprint 1 сущности из основного нижнего меню, но не ломать данные и повторно используемую логику.
7. Проверить сборку и вручную пройти сценарии Today.

## Архитектурные вопросы и варианты

### Вопрос 1. Что делать с нижней навигацией

Вариант A:
Оставить старые `AppTabId` и просто поменять подписи.

Плюсы:
- Быстрее.
- Меньше файлов.

Минусы:
- В коде `advisor` будет означать “Прогноз”, а `more` — “Настройки”.
- Дальше это усложнит Sprint 2 и поддержку.

Вариант B:
Ввести новые ids вкладок и обновить shell тонким слоем.

Плюсы:
- Чистая модель навигации.
- Меньше скрытого технического долга.
- Проще масштабировать дальше.

Минусы:
- Нужно изменить больше файлов прямо сейчас.

Выбор:
Вариант B. Он лучше масштабируется и не создаёт путаницу “старое имя / новый смысл”.

### Вопрос 2. Где считать состояние экрана “Сегодня”

Вариант A:
Считать всё прямо в `TodayScreen.tsx`.

Плюсы:
- Быстро.

Минусы:
- Экран станет ещё тяжелее.
- Труднее тестировать и развивать.

Вариант B:
Вынести расчёты в отдельный helper уровня `src/lib/today/*`.

Плюсы:
- Проще читать экран.
- Проще дорабатывать правила статуса дня, риска и главного действия.

Минусы:
- Появляется ещё один файл.

Выбор:
Вариант B. Это минимальное, но масштабируемое разделение UI и вычислений.

### Вопрос 3. Как внедрить “Добавить операцию”

Вариант A:
Оставить встроенный `VoiceRecorder` прямо на Today.

Плюсы:
- Минимум изменений.

Минусы:
- Это нарушает требование “внизу только одна главная кнопка”.
- Экран будет перегружен.

Вариант B:
Открывать существующий быстрый ввод в диалоге по кнопке.

Плюсы:
- Соответствует PRD.
- Сохраняет текущую логику быстрого ввода.
- Даёт чистый сценарий возврата на Today.

Минусы:
- Нужно слегка расширить `VoiceRecorder`.

Выбор:
Вариант B. Это лучший баланс между новым UX и переиспользованием текущего ввода.
